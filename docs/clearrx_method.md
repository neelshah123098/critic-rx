# ClearRx Method Notes

These notes describe the final ClearRx method used by the reproducibility
runner.

The submitted endpoint contract is pinned by:

- `prompts/formalrx_system_prompt.md`
- `prompts/formalrx_prompt_template.txt`
- `temperature=0`
- `max_tokens=1024`
- request timeout `300` seconds
- `stream=false`

## Diagnostic Protocol

ClearRx uses the CriticLeanGPT runtime as the final semantic critic. The row is
handled as a small independent diagnosis:

1. reconstruct the intended mathematical theorem from the informal statement;
2. reconstruct the theorem expressed by the Lean statement under the header;
3. compare the two meanings, not the surface strings;
4. if equivalent, output `aligned` and null/N/A diagnosis fields;
5. if different, route to the most specific SCI-28 category;
6. identify the smallest responsible Lean fragment;
7. produce the minimal corrected Lean statement;
8. check that verdict, category, span, and correction describe one mismatch.

## Why The Prompt Is Pinned

The GGUF runtime is packaged as a Dockerized FormalRx structured-diagnosis
endpoint. Small changes to prompt shape, streaming behavior, temperature, token
budget, or runtime path can cause the server to behave like an ordinary
text-generation model instead of the structured endpoint used for the
submission.

The runner therefore keeps the final user message in the exact FormalRx row
template, uses the runtime system prompt as the base, and appends only the
row-local evidence block generated by the premodel pipeline.

## Row-Local Premodel Evidence

The repository uses a deterministic premodel evidence pipeline. It does not
call an LLM and does not decide the final answer. It extracts row-local cues and
appends them to the system prompt as non-authoritative evidence.

The pipeline is:

1. `IntentLens` (`clearrx/premodel_intent.py`) extracts informal-statement cues:
   quantifiers, variables, domains, constraints, functions, operators,
   constants, concepts, and the conclusion surface.
2. `LeanLens` (`clearrx/premodel_lean.py`) extracts Lean-side cues:
   declaration kind/name, binders, hypotheses, variables, domains,
   constraints, operators, constants, target surface, and candidate spans.
3. `DeltaMap` (`clearrx/premodel_delta.py`) creates neutral semantic
   checkpoints such as missing constraints, operator changes, domain mismatch,
   or Lean-specific truncation risks.
4. `TaxonomyLens` (`clearrx/taxonomy_lens.py`) keeps the full FormalRx SCI-28
   taxonomy available every row, ranks likely categories from the checkpoints,
   and adds priority/disambiguation guidance.
5. `Evidence Composer` (`clearrx/premodel_evidence.py`) renders the final
   prompt add-on: SCI-28 checklist, dynamic likely-category evidence,
   row-local cues, and disambiguation rules.

The prompt makes each row explicitly account for all 28 FormalRx categories
without using a second generative model. It remains row-local and does not use
the test set as a retrieval corpus.

## Required Disclosure

No information from other levels, variants, reformulations, or related rows of
the same problem is used during inference. Each prediction is computed only
from the current row's header, informal statement, and candidate Lean formal
statement.

The submitted inference path uses the local Dockerized OpenAI-compatible
llama.cpp server for `criticleanGPT-Qwen3-8B-RL` as its only model endpoint. It
does not use external search engines, web search, remote LLM APIs, retrieval
databases, or another generative model during inference.

## Output Normalization

The runner performs only schema-preserving normalization before writing
`predictions.jsonl`:

- aligned verdicts force `error_category`, `error_segment`, and
  `corrected_statement` to JSON null;
- `N/A`, `NA`, `none`, and `null` string values are treated as null;
- known endpoint spelling aliases are mapped to the official SCI-28 label
  names, for example `Integration/Differentiation Confusion` to
  `Integration/Diff. Confusion`.

No prediction is changed using another test row or leaderboard feedback.
